# Greenfield social network app with Expo, Reanimated, and Vercel API stubs

A complete React Native social network shell built on Expo Router (file-based routing), featuring five tabs (feed, discover, create, messages, profile), a custom design system with coral/sage/cream/charcoal/gold tokens, dark/light theming persisted via MMKV, entrance animations via Reanimated layout transitions, and a set of Vercel serverless functions that return hardcoded mock data. The app uses Zustand for state management with no persistence layer for anything except theme mode, and never calls its own API endpoints â€” the client renders entirely from in-memory mock data.

**Watch for:** Authentication is completely bypassed on the client (login/register store mock user without validation or API call) â€” **confirmed**. The API endpoints have zero auth middleware (no token checks), meaning the deployed Vercel functions are open to unauthenticated requests â€” **confirmed**. Type mismatches between store models and shared types force `as unknown as` casts scattered across screens â€” **confirmed**. No test coverage exists â€” **confirmed**.

## High-level view

The design system is well-structured: a layered token file defines color scales, spacing, border radii, and typography; themed objects map those tokens to semantic roles (background, text, accent, border, status); a React context distributes the resolved theme. Dark mode is cleanly gated behind a single Zustand store with MMKV persistence.

The client never contacts the API layer. Every screen initializes from `mockData.ts` constants, making the Vercel functions dead code that cannot be validated through usage. The store types (e.g., `Post` in `feedStore.ts`) diverge from the shared types (e.g., `Post` in `src/types/index.ts`) â€” fields like `sharesCount`, `isBookmarked`, and `authorUsername` exist only in the shared version, forcing `as unknown as` casts across multiple screens.

Authentication is entirely decorative. The login screen stores a mock user without validation or API call, never persists a token, and no navigation guard in the root layout redirects unauthenticated users â€” a cold start drops directly into whatever route the deep link resolves to.

The Vercel API layer performs no authentication, authorization, or input sanitization beyond basic JSON parsing. The `POST /api/posts` handler spreads untrusted user input directly into the response object via `...postData`, which would be a stored injection vector in production.

Screens hardcode `paddingTop: theme.spacing['2xl']` (48px) instead of using `useSafeAreaInsets()` from the already-installed `react-native-safe-area-context`, which will clip or over-pad on devices with different notch sizes.

<summary>Details</summary>

## Disconnected API layer

None of the screen files import `fetch` or any HTTP client. The feed screen loads `mockPosts` from `src/utils/mockData.ts` into Zustand after an 800ms `setTimeout` delay (simulating a network call). The messages screen loads `mockConversations` the same way. The chat screen loads messages from the same file. The Vercel serverless functions under `api/` serve no runtime purpose â€” they exist purely as a contract sketch.

The API contract is already drifting. The API's post shape matches `src/types/index.ts` (with `sharesCount`, `authorUsername`) while the Zustand store's `Post` interface lacks those fields. When the fetch layer is eventually added, every screen that currently uses `as unknown as` casts will need rework.

## Authentication posture and navigation guard gap

The root layout (`app/_layout.tsx`) unconditionally renders the stack. There is no conditional redirect based on `useAuthStore().isAuthenticated`. A user who has never logged in can navigate directly to `/(tabs)` via deep link or cold start.

The login screen calls:

```ts
login({
  id: currentUser.id,
  username: currentUser.username,
  displayName: currentUser.displayName,
  avatar: currentUser.avatar,
  bio: currentUser.bio,
});
router.replace('/(tabs)');
```

No network request, no token stored, no error handling. The auth store has no persistence middleware (unlike the theme store which uses MMKV), so logging out and reopening the app lands the user in an unauthenticated state with no redirect logic to catch it.

On the API side, every endpoint processes requests without checking an `Authorization` header. The `POST /api/auth/login` handler returns a `mock-jwt-token-{timestamp}` that nothing ever validates.

The profile screen's `useEffect` that force-calls `login()` if `user` is null papers over the lack of persistence but creates a circular dependency where the profile screen is both a consumer and producer of auth state.

## Type system friction

Three separate `Post` interfaces exist:

1. `src/types/index.ts` â€” the canonical shared type with `sharesCount`, `isBookmarked`, `authorUsername`
2. `src/store/feedStore.ts` â€” a reduced version missing those fields
3. Inline in `api/posts/index.ts` â€” matches the shared type

The store barrel (`src/store/index.ts`) re-exports `Post` from `feedStore.ts`, so any consumer importing from `@/store` gets the reduced type. Screens resolve this by casting: `mockPosts as unknown as import('../../src/store/feedStore').Post[]`. The same pattern appears in `messages.tsx` with `Conversation` types.

`Conversation` in `chatStore.ts` lacks `participantUsername` and `isOnline` that exist in `src/types/index.ts`.

## PostCard filled-icon and double-tap detection

The like button renders `<Feather name="heart" ... />` regardless of `post.isLiked` â€” the ternary is `name={post.isLiked ? 'heart' : 'heart'}`, which is always the same icon. Differentiation relies solely on color, which is an accessibility concern for colorblind users. The same applies to the bookmark icon: `name={isBookmarked ? 'bookmark' : 'bookmark'}`.

The double-tap detection stores `lastTap` as a Reanimated shared value but reads and writes it on the JS thread (`handleDoubleTap` is a plain function, not a worklet). A `useRef<number>` would be semantically correct and avoid the `.value` indirection.

## Vercel API input handling

`POST /api/posts` uses:

```ts
const newPost = {
  id: 'p-' + Date.now(),
  authorId: 'current',
  // ...defaults
  ...postData,  // <-- untrusted spread
};
```

An attacker can override `authorId`, `id`, or inject arbitrary fields into the response. In a real backend with a database, this would be a stored injection vector. Even for a mock API, it sets a dangerous pattern that will carry forward.

The other POST handlers (`/api/posts/[id]/comments`, `/api/chats/[id]`) are more conservative â€” they destructure specific fields from the body.

## Hardcoded safe-area padding

Every screen uses `paddingTop: theme.spacing['2xl']` (48px) as a stand-in for safe-area insets. The project has `react-native-safe-area-context` installed but never uses `useSafeAreaInsets()` or `<SafeAreaView>`. This will clip content on devices with larger notches or dynamic islands and will over-pad on devices with no notch.
