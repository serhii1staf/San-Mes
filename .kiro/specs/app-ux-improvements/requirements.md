# Requirements Document

## Introduction

Набор UX-улучшений для мобильного приложения San (Expo/React Native + Supabase). Цель — исправить критические проблемы пользовательского опыта: потеря данных при переключении табов, отсутствие pinch-to-zoom в просмотрщике фото, некорректная логика редактирования поста (создаёт новый вместо обновления), неоптимальное расположение кнопки «Опубликовать» и возможная потеря качества фотографий при загрузке.

## Glossary

- **Приложение**: мобильное React Native приложение San-Mes, работающее на Expo
- **Tab_Navigator**: компонент навигации по вкладкам (Home, Search, Create, Messages, Profile)
- **Feed_Screen**: экран ленты постов (вкладка Home)
- **Profile_Screen**: экран профиля пользователя (вкладка Profile)
- **Create_Screen**: экран создания/редактирования поста (вкладка Create)
- **Image_Viewer**: модальное окно полноэкранного просмотра изображения
- **Zustand_Store**: глобальное хранилище состояния приложения (feedStore, authStore)
- **Supabase**: бэкенд-сервис для хранения данных и файлов
- **Post**: запись пользователя, содержащая текст и/или изображения
- **ImagePicker**: модуль expo-image-picker для выбора изображений из галереи или камеры

## Requirements

### Requirement 1: Сохранение данных при переключении табов

**User Story:** Как пользователь, я хочу чтобы данные не пропадали при переключении между вкладками, чтобы не ждать повторную загрузку каждый раз.

#### Acceptance Criteria

1. WHEN пользователь переключается между вкладками Tab_Navigator, THE Zustand_Store SHALL сохранять загруженные данные (посты, профиль, сообщения) в памяти без повторной загрузки с сервера
2. WHEN пользователь возвращается на Feed_Screen после переключения на другую вкладку, THE Feed_Screen SHALL отображать ранее загруженные посты в течение 300 миллисекунд без выполнения сетевого запроса к Supabase
3. WHEN пользователь возвращается на Profile_Screen после переключения на другую вкладку, THE Profile_Screen SHALL отображать ранее загруженные данные профиля и посты пользователя в течение 300 миллисекунд без выполнения сетевого запроса к Supabase
4. WHEN пользователь возвращается на ранее посещённую вкладку после переключения, THE Tab_Navigator SHALL сохранять позицию прокрутки списка в том же положении, в котором пользователь покинул вкладку
5. WHEN пользователь выполняет pull-to-refresh на Feed_Screen, THE Feed_Screen SHALL отображать индикатор обновления, загрузить свежие данные с сервера и обновить Zustand_Store
6. IF сетевой запрос при pull-to-refresh завершается ошибкой, THEN THE Feed_Screen SHALL прекратить отображение индикатора обновления, сохранить ранее загруженные данные на экране и показать сообщение об ошибке соединения

### Requirement 2: Pinch-to-zoom в просмотрщике фото

**User Story:** Как пользователь, я хочу приближать и перемещать фотографии в полноэкранном просмотрщике двумя пальцами, чтобы детально рассматривать изображения.

#### Acceptance Criteria

1. WHEN пользователь открывает Image_Viewer, THE Image_Viewer SHALL поддерживать жест pinch-to-zoom для масштабирования изображения от 1x до 3x
2. WHILE изображение масштабировано (zoom > 1x), THE Image_Viewer SHALL поддерживать жест панорамирования двумя пальцами, ограничивая перемещение так, чтобы край изображения не выходил за пределы границы viewport
3. WHEN пользователь выполняет двойной тап на изображении в Image_Viewer, THE Image_Viewer SHALL переключать масштаб между 1x и 2x, центрируя увеличение на точке касания
4. WHEN масштаб возвращается к 1x, THE Image_Viewer SHALL сбрасывать позицию панорамирования к исходному центрированному положению
5. IF пользователь масштабирует изображение за пределы максимального значения 3x, THEN THE Image_Viewer SHALL вернуть масштаб к значению 3x с анимацией длительностью не более 300 мс
6. IF пользователь масштабирует изображение ниже минимального значения 1x, THEN THE Image_Viewer SHALL вернуть масштаб к значению 1x с анимацией длительностью не более 300 мс

### Requirement 3: Корректное редактирование поста

**User Story:** Как пользователь, я хочу чтобы при нажатии «Опубликовать» в режиме редактирования обновлялся существующий пост, а не создавался новый.

#### Acceptance Criteria

1. WHEN пользователь нажимает «Опубликовать», IF editingPostId содержит идентификатор поста, THEN THE Create_Screen SHALL вызывать supabase.from('posts').update() с полями content и image_url для поста с указанным идентификатором
2. WHEN редактирование поста завершено успешно, THE Create_Screen SHALL найти пост по editingPostId в кэше AsyncStorage (ключи @san:feed_posts и @san:my_posts) и заменить его данные обновлёнными значениями, не добавляя новый элемент
3. WHEN пользователь открывает пост для редактирования, THE Create_Screen SHALL заполнить текстовое поле значением content из editingPost и отобразить существующие изображения (imageUrl/imageUrls) в области превью изображений
4. IF обновление поста завершается с ошибкой от Supabase, THEN THE Create_Screen SHALL отобразить Alert с сообщением об ошибке и сохранить текст и выбранные изображения пользователя в форме без очистки
5. WHEN редактирование поста завершено успешно, THE Create_Screen SHALL сбросить editingPostId в null и выполнить навигацию назад (router.back) на предыдущий экран
6. WHEN пользователь редактирует пост и изменяет изображения, IF изображение является новым локальным файлом (не URL), THEN THE Create_Screen SHALL загрузить его через uploadPostImage перед вызовом update; IF изображение является существующим удалённым URL, THEN THE Create_Screen SHALL сохранить этот URL без повторной загрузки

### Requirement 4: Перемещение кнопки «Опубликовать»

**User Story:** Как пользователь, я хочу видеть кнопку «Опубликовать» в правом верхнем углу экрана (как в Instagram/Telegram), чтобы интерфейс был привычным и удобным.

#### Acceptance Criteria

1. THE Create_Screen SHALL отображать кнопку «Опубликовать» в правой части навигационного хедера (headerRight) на уровне заголовка экрана
2. WHILE содержимое поста пустое (нет текста, нет изображений и нет данных репоста), THE Create_Screen SHALL отображать кнопку «Опубликовать» в неактивном (disabled) состоянии с непрозрачностью (opacity) 0.4
3. THE Create_Screen SHALL NOT отображать кнопку публикации в нижней области контента
4. THE Create_Screen SHALL отображать кнопку «Опубликовать» текстовым стилем без фонового контейнера, используя цвет accent.primary в активном состоянии
5. WHILE идёт процесс публикации, THE Create_Screen SHALL отображать кнопку «Опубликовать» в неактивном (disabled) состоянии с индикацией загрузки

### Requirement 5: Сохранение качества фотографий при загрузке

**User Story:** Как пользователь, я хочу чтобы мои фотографии загружались без потери качества, чтобы публикации выглядели так же хорошо, как оригиналы.

#### Acceptance Criteria

1. WHEN пользователь выбирает изображение из галереи через ImagePicker, THE ImagePicker SHALL использовать параметр quality со значением 1.0 (максимальное качество) и не применять масштабирование (allowsEditing: false)
2. WHEN пользователь делает фото камерой через ImagePicker, THE ImagePicker SHALL использовать параметр quality со значением 1.0 (максимальное качество) и не применять масштабирование (allowsEditing: false)
3. THE uploadPostImage SHALL передавать изображение на сервер в оригинальном разрешении без дополнительного сжатия или изменения размера на стороне клиента, при условии что размер файла не превышает 20 МБ
4. IF размер выбранного изображения превышает 20 МБ, THEN THE Приложение SHALL отобразить сообщение об ошибке с указанием максимально допустимого размера файла и не выполнять загрузку
5. IF загрузка изображения на Supabase Storage завершается неудачей, THEN THE Приложение SHALL отобразить сообщение об ошибке с указанием причины сбоя и сохранить введённый контент поста без потери данных
6. WHEN изображение загружено на Supabase Storage, THE Приложение SHALL отображать его в ленте с использованием resizeMode="cover" в контейнере высотой 300 логических пикселей с сохранением исходного соотношения сторон

### Requirement 6: OTA-деплой после реализации

**User Story:** Как разработчик, я хочу чтобы все UX-улучшения были доставлены пользователям через OTA-обновление (eas update), чтобы изменения стали доступны без пересборки приложения.

#### Acceptance Criteria

1. WHEN все изменения закоммичены и запушены в ветку main, THE GitHub Actions workflow (ota-update.yml) SHALL автоматически запустить eas update --branch production --platform ios
2. WHEN OTA-обновление опубликовано успешно, THE Приложение SHALL загрузить новую версию бандла при следующем запуске без необходимости обновления через App Store
3. IF OTA-сборка завершается с ошибкой в GitHub Actions, THEN THE workflow SHALL завершиться с ненулевым exit code и отобразить ошибку в логах Actions
4. WHEN выполняется push в main с UX-улучшениями, THE commit message SHALL содержать описание внесённых изменений для поля message в eas update
